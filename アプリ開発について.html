Xcodeをインストール

Mac App StoreからXcodeをインストール。20GB以上の空き容量が必要。

export PATH="$PATH:/path/to/flutter/bin"

CocoaPodsをインストール

iOSの依存関係管理ツール。

sudo gem install cocoapods
⚠️ 注意： Xcode 16以降は CocoaPods 1.15+ が必要。バージョンを確認すること。
4
flutter doctorで確認 
flutter doctor

全項目に ✅ がつけばOK。Xcodeのライセンス同意が必要な場合あり。

7. エラー集（実体験ベース）

実際の開発で遭遇したエラーと解決方法をまとめています。

❌ 1. Bundle ID不一致

症状： ビルドは通るが、FamilyControlsなどのCapabilityが機能しない

✅ 解決法： Xcode → Runner → Signing & Capabilities のBundle IdentifierとApple Developer PortalのBundle IDを完全一致させる。大文字小文字も注意。

❌ 2. Build Cycle Error

症状： "Cycle inside Runner; building could produce unreliable results"

✅ 解決法： Build Phases の順序を変更。「Embed Foundation Extensions」を「Thin Binary」より前に移動する。

❌ 3. シミュレーターで白画面

症状： シミュレーターでアプリを起動すると画面が真っ白のまま

原因1： objective_c.framework がiOSデバイス用でシミュレーター非対応

原因2： FamilyControls/DeviceActivity APIをシミュレーターで呼んでいる

✅ 解決法：
  1. flutter clean && flutter pub get
  2. DerivedDataを削除
  3. XcodeのGUIではなく flutter run -d [シミュレーターID] で実行
  4. FamilyControls系は #if !targetEnvironment(simulator) で囲む

❌ 4. Distribution時にEntitlementエラー

症状： "Provisioning profile doesn't include the Family Controls capability"

✅ 解決法： ExtensionのEntitlementsからFamilyControlsを削除する。FamilyControlsはメインアプリのEntitlementsにだけ必要。

❌ 5. コールドスタートクラッシュ

症状： デバッガーなしで起動するとクラッシュ/白画面

✅ 解決法： main.dart の初期化処理をtry-catchで囲む。 
try {
  await PremiumService.instance.init();
} catch (e) {
  debugPrint('init error: $e');
}

❌ 6. タスクを切るとアプリが落ちる

症状： アプリスイッチャーでアプリを閉じると再起動できない

✅ 解決法： Xcode → Edit Scheme → Run → Build Configuration を Release に変更し、Debug executable をOFF にする。

❌ 7. 進捗バーが途中で止まる

症状： 制限時間が短い場合（1〜3分）に進捗バーが50%や75%で止まる

✅ 解決法： 閾値（25/50/75/100%）が同じ分数になる場合に重複が発生する。Setで重複排除し、100%は必ず含めるロジックを実装する。

❌ 8. dSYM警告

症状： "The archive did not include a dSYM for objective_c.framework"

✅ 解決法： 無視してOK。審査には影響しない。クラッシュレポートの一部が読めなくなるだけ。

❌ 9. CocoaPods object version エラー

症状： "Unable to find compatibility version string for object version 70"

✅ 解決法： gem install cocoapods で最新版に更新。それでも失敗する場合は flutter run コマンドを使う（Flutterが自動対応してくれる）。

遭遇したエラーと解決策

❌ RLS ポリシーでペアリングができない

招待コードを入力しても「無効なコード」エラーになる。他人が作ったコードをSELECTできないRLSポリシーが原因。

✅ 解決： invite_codes_redeem_select ポリシーを追加。used = false AND expires_at > now() の条件で認証済みユーザーなら参照可能に変更。
また couples_insert に user_b_id = auth.uid() 条件のINSERT許可ポリシーも必要。

❌ generate-advice が常に 401 Unauthorized

DBトリガーからEdge Functionを呼び出すと401。WEBHOOK_SECRET環境変数の値とDBトリガーが送るシークレットが不一致だった。

✅ 解決： Edge Function内の WEBHOOK_SECRET チェックを削除。DBトリガーからの内部呼び出しは関数URLが秘匿情報として機能するため、シークレット検証なしで問題なし。
verify_jwt: false かつWEBHOOK_SECRET不要に変更。

❌ Gemini API キー 429 (limit: 0)

APIキーを設定したのに429エラー。エラー詳細に limit: 0 と表示される。

✅ 解決： Google Cloud Console経由のキーは課金プロジェクトのため無料枠(limit)が0になる。Google AI Studio（aistudio.google.com）で「Create API key in new project」からキーを発行し直すこと。AI Studioのキーは1日1500リクエスト無料で使える。

❌ 提出後にホーム画面の状態が更新されない

SubmissionEditViewを閉じてもホームの「提出済み」フラグが変わらない。

✅ 解決： NotificationCenter を使って提出完了通知を発行。ホーム画面が .onReceive で受信して vm.fetch() を呼ぶ方式に。
extension Notification.Name { static let submissionCompleted = ... }

❌ シミュレーターでApple Sign In・Google Sign Inが動かない

AppleはシミュレーターのiCloud設定制限。GoogleはFirebase iOSキーはSupabase OAuth対応外。

✅ 解決： #if DEBUG ブロックでメール/パスワードログインを追加。
Supabaseの無料枠レートリミットでメール登録もできない場合は、SQL直接実行で auth.users にINSERT（email_confirmed_at を now() に設定）。

❌ コード入力がフリック入力のみになる

招待コード入力のTextFieldでアルファベット/数字キーボードが出ない。

✅ 解決： .keyboardType(.asciiCapable) と .autocorrectionDisabled() を追加。さらにOnChangeで .filter { $0.isLetter || $0.isNumber } でフィルタリング。

❌ Age Rating Declaration の型エラー（App Store Connect API）

API経由でAge Rating設定時に400エラー。messagingAndChat や userGeneratedContent をString型で送っていた。

✅ 解決： これらのフィールドはBoolean型。"messagingAndChat": true のように設定。また advertising・parentalControls・ageAssurance も必須フィールドのため含める。

月間レートリミット実装

無料枠でのコスト管理として、カップルごとに月5ラウンド制限を実装。

// Edge Function内での月間チェック

if (thisMonthAdvices.length >= FREE_ROUNDS_PER_MONTH) {
  // 両者にAPNs通知して終了
  return new Response("Rate limit reached", { status: 200 });
}

App Store Connect API キー（.p8）、APNs 鍵、Supabase Service Role キーなど、流出すると即座に悪用されうる秘密情報の取り扱いルールです。実際に AI アシスタントの対話ログに秘密鍵が平文で残っていた事例があり、そこから得た教訓をまとめています。

大原則

🔒
権限は 600

所有者のみ読み書き可

🚫
全文 dump 禁止

cat は使わない

🔁
漏洩即ローテ

疑いありなら revoke

📝
ログ監査

AI ログ・履歴を定期確認

秘密ファイルのパーミッション

秘密鍵ファイルは必ず所有者のみ読み書き可（600）にする。644 のままだと同一マシンの他ユーザーや、マルウェアに権限昇格されたプロセスが読める。

# 対象ファイルを一括で 600 に
chmod 600 ~/.blitz/asc-credentials.json
chmod 600 ~/.ssh/id_*
chmod 600 ~/AuthKey_*.p8

# 確認
ls -la ~/.blitz/asc-credentials.json
# → -rw------- が正常

❌ アンチパターン：credential ファイルを cat する

cat ~/.blitz/asc-credentials.json の出力は、シェル履歴・ターミナルスクロールバッファ・AI アシスタントの対話ログなど複数の場所に平文で残る。一度でも実行すれば「そのファイルは漏洩したものとして扱う」ぐらいの警戒が必要。

✅ 代わりに： 構造だけ知りたいならメタ情報のみ抽出する。 
python3 -c "
import json, pathlib
d = json.loads(pathlib.Path('~/.blitz/asc-credentials.json').expanduser().read_text())
print('keys:', list(d.keys()))
print('privateKey length:', len(d['privateKey']))
print('first line:', d['privateKey'].splitlines()[0])"

AI アシスタント利用時の注意

⚠️ 最重要： Claude Code / Cursor / Cline などの AI コーディングアシスタントは、bash ツールの stdout をそのまま会話ログ（JSONL 等）に保存する。cat で鍵を表示させると そのログファイルに鍵が永続化 される。ログの保存場所は ~/.claude/projects/, ~/.cursor/ など。
行為 リスク 代替
cat secret.json を AI に実行させる ログ・キャッシュに平文残存 長さ・先頭行・フィールド名のみ表示
鍵を会話にコピペして「これ使って」 会話全体がクラウドに送信・保存 ローカルファイルから読む script を書かせる
env var に鍵を設定してスクショを貼る OCR / キャプチャログで流出 コマンド名だけ見せる、値はマスキング
「動かないので rm -rf 全部見せて」 .env や .p8 が混入 対象ディレクトリを限定、--exclude

漏洩時の対応手順

revoke] B --> C[新しいキーを発行] C --> D[新キーをローカルに配置

コミットされていたか？} G -->|YES| H[履歴からも削除
git filter-repo
+ force push] G -->|NO| I[ローカル対応で完了] H --> J[事後監査
ASC Activity 等で
不審アクセス確認] I --> J style A fill:#FF3B30,color:#fff style B fill:#FF9500,color:#fff style J fill:#34C759,color:#fff
1
ASC API キーの失効（revoke）

App Store Connect → Users and Access → Integrations → App Store Connect API → 該当キーの「Revoke」。失効は即時反映、以降 JWT 署名が拒否される。自分が今使っているキーを revoke すると即ロックアウトされるので、作業用の別キーが必要な場合は先に新キーを発行しておく。

2
APNs 鍵の失効

Apple Developer → Certificates, Identifiers & Profiles → Keys → 対象キーの Revoke。プッシュ通知を送信中のサーバーがあれば新キーへの切り替えを忘れずに。

3
Supabase Service Role キーのローテーション

Supabase Dashboard → Project Settings → API → Service Role key の「Regenerate」。Edge Functions の環境変数も更新が必要（supabase secrets set）。

4
ログ・痕跡の削除

AI セッションログ、シェル履歴、CI のログなどから漏洩値を検索・削除する。

# Claude Code のセッションログから検索
grep -l "BEGIN PRIVATE KEY" ~/.claude/projects/*/*.jsonl

# 該当ファイルを削除
rm ~/.claude/projects/<project>/<session-id>.jsonl

# zsh / bash 履歴から削除（該当行を特定して）
history | grep -n "cat.*credentials"
# → 行番号を確認して ~/.zsh_history を直接編集
5
Git 履歴からの削除（公開リポジトリにコミットしてしまった場合） 
# 最新コミットのみなら amend
git rm --cached leaked-file.p8
git commit --amend --no-edit
git push --force-with-lease

# 過去のコミットにも入っているなら git filter-repo（要インストール）
git filter-repo --path leaked-file.p8 --invert-paths
git push --force-with-lease --all

Force push 後も GitHub のキャッシュから一定期間取得可能なので、結局 revoke が唯一の真の対策。履歴削除は補助的な作業。

予防策：そもそも漏らさない設計

リソース 推奨保管場所 コードからのアクセス方法
ASC API 鍵（.p8） ~/.blitz/, ~/.appstoreconnect/（600） ランタイムに open() で読む。環境変数でパスだけ渡す
APNs 鍵 Supabase Secrets / AWS Secrets Manager Edge Function 内で Deno.env.get()
Gemini / OpenAI API キー Supabase Secrets（クライアント側には置かない） 必ず Edge Function 経由で呼び出す
Supabase Service Role サーバー側環境変数のみ 絶対にクライアント（iOS アプリ）に埋め込まない
Firebase Admin 鍵 サーバー側環境変数のみ クライアントは公開 API キーのみ使用

❌ よくあるミス：iOS アプリのバイナリに秘密キーを埋め込む

クライアント配布バイナリは strings コマンドや ipa 解凍で中身が読める。Supabase Service Role キーや OpenAI キーなどを直接埋めると即流出。

✅ 解決： クライアントには Anon Key / Publishable Key（RLS で保護）のみ。機密操作は必ず Edge Function / バックエンドを経由させる。

定期監査のチェックリスト

🗓️ 四半期ごと
  • ASC API キーの使用状況確認
  • 古い .p8 ファイルの棚卸し
  • 不要になったキーは revoke
⏱️ 毎回
  • AI に credential 扱わせる前に一瞬考える
  • cat ではなくメタ情報抽出
  • スクショ前にマスキング
🚨 漏洩疑惑時
  • まず revoke（調査は後）
  • ASC Activity / 監査ログ確認
  • 不審な API アクセスなし？
💡 格言： 「漏洩したかも」と思った時点で、もう漏洩したものとして扱う。revoke のコストは数分。悪用された場合のコストは数百時間〜。迷ったら revoke。

Phase 1：Claude Design でアセット先行準備

💡 ポイント： UI とアセットを先に固めてから実装に入ると、後工程での手戻りが激減する。TapSchool では先にアイコン・スクショ構成・画面レイアウトを Claude Design で固めてから Swift 実装に入った。
1
アイコン草案を 5〜10 パターン生成

Claude Design / Canva MCP で「アプリの核となる動詞・名詞」を指定して複数案生成。丸型・角丸・グラデ有無などバリエーションを用意。人間はこの中から選ぶだけ。

2
主要画面の UI カンプ

ホーム・設定・メイン機能画面のカンプを Claude Design で起こす。SwiftUI での実装指示書として使える。カラーパレットをここで固定すると以降ブレない。

3
App Store スクショの「枠」を先に作る

キャッチコピー + 背景 + デバイスフレームの Canva テンプレを 5〜6 枚分用意。中身のアプリ画面は後で差し込むだけの状態にしておく。

Phase 2：Claude Code による実装

1
プロジェクト雛形を生成

XcodeBuildMCP の scaffold_ios_project で SwiftUI プロジェクトを新規作成。Bundle ID・Team・Deployment Target を指定すると Xcode が開ける状態で出てくる。

2
Phase 1 のカンプを実装指示として渡す

「このカンプ通りに SwiftUI で」で済む。配色・余白・フォントサイズを聞いてくるようなら、Phase 1 の時点で確定し切れていないサイン。

3
PrivacyInfo.xcprivacy を最初に入れる

データ収集しないアプリでも「収集していない」ことを宣言する必要がある。UserDefaults を使うなら NSPrivacyAccessedAPIType CA92.1 理由を記載。

<key>NSPrivacyTracking</key><false/>

Phase 3：XcodeBuildMCP でテスト・スクショ撮影

⚠️ 事前設定： UI 自動化（tap / swipe / snapshot_ui）はデフォルト無効。XCODEBUILDMCP_ENABLED_WORKFLOWS 環境変数に ui-automation を追加して有効化する（公式 Configuration）。
1
セッションデフォルトを設定

毎回 project / scheme / simulator を指定するのは冗長。session_set_defaults で一度設定しておくと以降 build_run_sim が引数ほぼ空で動く。

2
シミュレータで golden path を実行

build_run_sim → snapshot_ui で画面ツリー取得 → tap/type_text で操作 → screenshot で記録。

ただし App Store 用の生スクショだけは人間が撮るのが無難。UI 自動化で撮るとアニメーション中の微妙なフレームを拾うなどコントロールしづらい。

Phase 4：Canva + Pillow でスクショ仕上げ

1
生スクショを Canva テンプレに流し込む

Phase 1 で用意した 5〜6 枚のテンプレに、人間が撮った生スクショを差し込み、Canva MCP export-design で PNG 書き出し。

2
ASC 要求解像度にリサイズ（Pillow）

iPhone 6.9 inch 用は 1320×2868（APP_IPHONE_67 display type が受け付ける最大サイズ）。アスペクト比が合わなければ背景色でパディング。

from PIL import Image
from pathlib import Path
SRC = Path("./AppStore")
TW, TH = 1320, 2868
for i in range(1, 7):
    img = Image.open(SRC / f"promo_{i}.png").convert("RGB")
    scale = TW / img.size[0]
    new_h = int(round(img.size[1] * scale))
    scaled = img.resize((TW, new_h), Image.LANCZOS)
    bg = img.getpixel((5, 5))  # 左上の背景色で塗る
    canvas = Image.new("RGB", (TW, TH), bg)
    canvas.paste(scaled, (0, (TH - new_h) // 2))
    canvas.save(SRC / f"promo_{i}_1320x2868.png", "PNG")

Phase 5：ASC API でメタデータ・スクショ自動投入

💡 ポイント： ASC MCP は一部の GET_COLLECTION を禁じているため、スクショ upload のような複合操作は直接 REST API を叩く方が速い。JWT（ES256）を自作すれば認証は簡単。
1
JWT 発行（ES256） 
import jwt, time, json, pathlib
creds = json.loads(pathlib.Path("~/.blitz/asc-credentials.json").expanduser().read_text())
token = jwt.encode(
    {"iss": creds["issuerId"], "iat": int(time.time()),
     "exp": int(time.time())+1200, "aud": "appstoreconnect-v1"},
    creds["privateKey"], algorithm="ES256",
    headers={"kid": creds["keyId"], "typ": "JWT"},
)
2
スクショ upload は 3-step

（a）POST /v1/appScreenshots で予約 → uploadOperations を受け取る。（b）各 op の URL に binary を PUT。（c）PATCH で uploaded=true + sourceFileChecksum（MD5）を送信。

ハマり所：APP_IPHONE_69 という display type は存在しない。iPhone 6.9" 用でも APP_IPHONE_67 を使い、1320×2868 の PNG を投入する。

3
冪等性：既存 set を再利用 or クリア

同じ display type の set を重複作成しようとすると 409。既存を探して再利用、その中の screenshot は削除してから再投入する。

Phase 6：blitz で検証 → 提出

1
メタデータ抜け漏れチェック

blitz（ASC MCP を束ねる橋渡し）で「What's New」「アプリ説明」「キーワード」「サポート URL」「プライバシーポリシー URL」「年齢制限」「カテゴリ」「価格」「Build 添付」などをまとめて確認。抜けがあれば埋める。

2
プライバシー栄養ラベルは ASC Web で手動

この項目は API が公開されていない。初回のみ人間が Web UI で設定。データ収集なしのアプリなら「Data Not Collected」を選ぶだけで済む。

3
Archive は Xcode（or xcodebuild）で

Xcode → Product → Archive。CI から流したいなら xcodebuild archive -exportArchive + API Key で Transporter 経由アップロード。TapSchool では手動 Archive を採用（年に数回しか回さないので投資効果が薄い）。

4
人間最終レビュー → Submit for Review

ASC Web UI で表示を一通り眺め、スクショ・説明文・価格・審査向けメモに違和感がないことを確認して提出。
